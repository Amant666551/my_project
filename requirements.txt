Let me read through the code carefully before diagnosing and diagramming.

Here's a thorough analysis of your codebase. There are two distinct problem classes: voice quality loss in TTS and pipeline latency. Let me break both down, then propose an optimized architecture.

Problem 1: TTS loses voice characteristics
The root issue is in tts.py and main.py. You're using edge-tts, a Microsoft Azure TTS wrapper — it produces consistent voice output, but there are two bugs that silently degrade quality:

Bug 1 — asyncio.run() inside a sync loop: In main.py, you're calling asyncio.run(play_audio(text)) inside a blocking while True. This creates and destroys a new event loop every iteration. Edge-TTS reuses an underlying HTTPS session; repeated event loop teardown can interrupt mid-stream audio chunks, causing the saved MP3 to be truncated or have silent gaps. The result sounds like clipped or robotic speech.

Bug 2 — temp file race condition: The audio is written to a temp file, then immediately loaded by pygame before the OS flushes the write buffer. On some systems (especially Docker containers), pygame.mixer.music.load() reads a partially written file.

Fix: Use a single persistent asyncio event loop throughout the process lifecycle, and use asyncio.Event to signal playback readiness rather than relying on file existence.

Problem 2: Pipeline latency
Your current flow is:

Microphone → ASR (Whisper) → Redis pub → MT (Qwen 1.8B) → Redis pub → TTS → Speaker
The bottlenecks are:

ASR: faster-whisper small on CPU with int8 takes ~1.5–3s per utterance. You're also writing to a temp WAV file on every utterance, adding I/O overhead.

MT: Qwen 1.8B with FP16 auto device mapping and max_new_tokens=256 is very slow on CPU — easily 5–15 seconds. The translate() method signature also has a mismatch: api.py calls translator.translate(text, source_lang=..., target_lang=...) but translator.py's translate() only accepts text — the other args are silently ignored and will cause a TypeError in strict mode.

Redis hop overhead: Three separate pub/sub channels add minor latency but more importantly they add complexity with no backpressure — if MT is slow, queued Redis messages pile up and you get an ever-growing delay between what was said and what is played.

The asr_client.py is empty — this means there's no clean client abstraction for ASR the way there is for MT and TTS.

import asyncio, threading

async def _play(text):

future = asyncio.run_coroutine_threadsafe(_play(text), loop)

Here's a clear picture of what changed and how to get your voice working:

Why edge-tts can never clone your voice
It's a cloud API that sends your text to Microsoft's servers and returns audio from a pre-trained fixed speaker. There's no mechanism to inject a custom voice — the voice identity is locked inside Microsoft's infrastructure. No amount of configuration can change this.

The two practical options for your setup
XTTS-v2 is the better choice if quality matters most. It's a single model that takes your reference audio clip and conditions the entire synthesis on it — pitch, timbre, accent, and rhythm are all transferred. The tradeoff is speed: on CPU it takes ~6–10 seconds per sentence, which may introduce a noticeable lag in real-time interpretation.

OpenVoice v2 is the better choice if speed matters more. Its two-stage design (MeloTTS generates clean speech, then a tone-color converter transplants your vocal timbre onto it) runs in ~2–4 seconds on CPU. Timbre is very well preserved; prosody and rhythm are slightly less faithful than XTTS.

What makes a good reference clip
The cloning quality depends almost entirely on the reference audio. The recorder script (record_voice.py) shows a live level meter and warns if the signal is too quiet. The key things: record in a quiet room, speak for at least 15 seconds, and avoid any music or echo in the background. A noisy reference will produce noisy cloned speech — the model can't separate your voice from background noise.
