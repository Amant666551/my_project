四人录制执行版 v1

用途

- 用于生成闭集实时多说话人语音翻译评测数据
- 供 4 位组员按固定顺序录制
- 对应参考集文件：eval/reference_set_4spk_v1.jsonl


角色与路由编号

- A 号说话人：spk_a -> 001
- B 号说话人：spk_b -> 002
- C 号说话人：spk_c -> 003
- D 号说话人：spk_d -> 004


录制要求

1. 每句单独说，不要把相邻两句连成一长句。
2. 保持自然语速，不要故意一字一顿。
3. 短句场景按短句说，长句场景按正常口语说，不要统一拉长。
4. 不要在句首句尾额外加“嗯”“啊”“那个”之类未写出的口头词。
5. 如果是连续切换场景，相邻两位说话人之间停顿尽量短。
6. 如果一条读错，直接重读该条，不要跳号。


全局录制顺序

Session 01 Warmup

001 A 你好，可以听到我说话吗？
002 A 这次我的声音正常吗？
003 B 可以，我这边听起来没问题。
004 A 那我再说长一点，看看会不会更稳定。
005 C 现在轮到我说一句，你那边能分出来吗？
006 D 我只说一个短句。
007 D 然后马上再补一句。
008 B 刚才那个切换很快，系统可能比较容易抖。

Session 02 Meeting

009 A 今天下午三点我们要开项目例会。
010 B 会议室还是之前那个吗？
011 A 对，还是五楼靠窗的那间。
012 C 我可能会晚到十分钟，你们先开始。
013 D 如果投影仪再出问题，就改成线上。
014 B 我先把文档和议程发到群里。
015 C 可以，尤其把预算那一页单独标出来。
016 A 好，那下午见。

Session 03 Technical

017 B 这个接口昨天还能用，今天怎么就超时了？
018 A 我怀疑是鉴权服务那边响应变慢了。
019 D 日志里有没有明显的五百错误？
020 C 我刚查了一下，主要是重试次数突然变多了。
021 B 那先把超时时间调高两秒试试。
022 A 可以，但也要记录调整前后的延迟变化。
023 D 我这边先拉一份最近三十分钟的日志。
024 C 行，等你拉完我们再一起看。

Session 04 Short Utterance

025 D 嗯。
026 D 嗯嗯。
027 A 你刚才是在表示同意吗？
028 D 对，我只是想测试这种特别短的句子。
029 B 短句往往最容易让说话人路由不稳定。
030 C 所以这个场景一定要单独统计。
031 A 那我再补一句稍长一点的。
032 A 如果这句能稳住，说明时长确实有帮助。

Session 05 Fast Switch

033 C 我先说一句，然后你马上接。
034 B 好，我现在立刻接上。
035 C 再换回来，看看系统会不会抖动。
036 D 我也插一句，增加一点干扰。
037 A 如果这里还能保持稳定，那就说明策略有效。
038 B 这类连续切换最能暴露 false switch 问题。
039 C 尤其是句子很短的时候。
040 D 那我们最后再做一次四人轮流发言。

Session 06 Round Robin

041 A 第一位先确认网络和麦克风状态。
042 B 第二位确认翻译方向和输出语言。
043 C 第三位确认当前音色是否已经正确激活。
044 D 第四位确认日志文件是否正在正常写入。
045 A 如果这些都正常，就开始正式测试。
046 B 正式测试时，记得同时关注准确率和时延。
047 C 还要看同一个人多轮说话时 route 会不会漂。
048 D 最后把结果导出来，再统一计算指标。

录制后建议

1. 保留每句的编号，不要改。
2. 如果后面要人工标注或核对系统输出，优先按编号对齐。
3. 如果某位组员临时换人，保持该人的 route 编号不变，只记录实际更换情况。
